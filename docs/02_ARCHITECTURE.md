# 02_ARCHITECTURE.md

# Архитектура Directory Platform

## Назначение документа

Этот документ описывает архитектуру Directory Platform, её основные модули, уровни ответственности и правила взаимодействия между ними.

Документ отражает целевую архитектуру платформы. Реализация отдельных модулей может происходить постепенно.

---

# Общая концепция

Directory Platform — это универсальная система для создания тематических каталогов.

Платформа состоит из общего ядра и отдельных проектных модулей.

Общее ядро содержит функциональность, которая может использоваться повторно в разных каталогах.

Проектный модуль содержит функциональность, относящуюся только к конкретному каталогу.

---

# Общая структура

```text
Directory Platform

├── Platform Core
│   ├── GeoCore
│   ├── DataCore
│   ├── DirectoryCore
│   ├── SEOCore
│   ├── ImportCore
│   ├── MediaCore
│   ├── SearchCore
│   └── ReviewCore
│
└── Project Modules
    ├── PflegeIndex
    ├── BestatterIndex
    ├── FriedhofIndex
    ├── ÄrzteIndex
    └── другие каталоги
```

---

# Platform Core

Platform Core — общее ядро платформы.

Его модули не должны зависеть от специфики одного конкретного каталога.

Например, GeoCore не должен содержать понятия Pflegeheim, Pflegedienst или Bestatter.

---

# GeoCore

## Назначение

GeoCore отвечает за географическую структуру данных и географическую навигацию.

## Базовая иерархия

```text
State
→ District
→ City
→ Facility
```

## Ответственность GeoCore

GeoCore отвечает за:

* федеральные земли;
* районы;
* города;
* географические связи между объектами;
* географические URL;
* географические страницы;
* внутреннюю навигацию между уровнями географии.

## Пример для Brandenburg

```text
Brandenburg
→ Landkreis Dahme-Spreewald
→ Luckau
→ конкретное учреждение
```

GeoCore должен быть универсальным и пригодным для любого каталога.

---

# DataCore

## Назначение

DataCore отвечает за структуру и хранение данных объектов каталога.

## Типичные данные объекта

* название;
* описание;
* адрес;
* телефон;
* электронная почта;
* сайт;
* координаты;
* статус;
* тип объекта;
* связанные услуги;
* дата обновления данных.

## Основной принцип

DataCore хранит общую структуру объекта.

Специфические поля конкретного каталога должны находиться в проектном модуле или в расширяемой системе атрибутов.

---

# DirectoryCore

## Назначение

DirectoryCore отвечает за общую логику каталога.

## Ответственность

* страницы списков;
* страницы объектов;
* категории;
* фильтрация;
* пагинация;
* сортировка;
* связи между объектами;
* похожие объекты;
* общие компоненты карточек.

DirectoryCore не должен содержать бизнес-логику только одного проекта.

---

# SEOCore

## Назначение

SEOCore отвечает за единые правила поисковой оптимизации.

## Ответственность

* title;
* meta description;
* canonical URL;
* Open Graph;
* структурированные данные Schema.org;
* XML Sitemap;
* breadcrumbs;
* правила индексации;
* шаблоны SEO-текстов;
* предотвращение дублирования страниц.

## Основной принцип

SEO не должно реализовываться отдельно и несогласованно на каждой странице.

Общие механизмы должны находиться в SEOCore.

---

# ImportCore

## Назначение

ImportCore отвечает за загрузку, преобразование и проверку внешних данных.

## Ответственность

* импорт CSV;
* импорт JSON;
* импорт данных из API;
* нормализация;
* проверка обязательных полей;
* поиск дубликатов;
* журналирование ошибок;
* предварительный просмотр импорта;
* повторный запуск импорта.

ImportCore должен быть отделён от публичной части сайта.

Ошибка импорта не должна нарушать работу каталога.

---

# MediaCore

## Назначение

MediaCore отвечает за изображения и другие медиафайлы.

## Ответственность

* загрузка изображений;
* хранение;
* изменение размера;
* оптимизация;
* форматы изображений;
* изображения по умолчанию;
* привязка медиа к объектам;
* проверка источника и прав использования.

---

# SearchCore

## Назначение

SearchCore отвечает за поиск и расширенную фильтрацию.

## Ответственность

* полнотекстовый поиск;
* подсказки;
* поиск по городу;
* поиск по категории;
* фильтрация по атрибутам;
* сортировка результатов;
* обработка пустых результатов.

На ранних этапах поиск может использовать стандартные возможности базы данных.

Отдельный поисковый сервис должен внедряться только при реальной необходимости.

---

# ReviewCore

## Назначение

ReviewCore отвечает за отзывы, оценки и пользовательскую обратную связь.

## Ответственность

* создание отзывов;
* рейтинг;
* модерация;
* защита от спама;
* ответы организаций;
* жалобы на отзывы;
* публикация и скрытие отзывов.

ReviewCore не должен внедряться до определения правил модерации и юридических требований.

---

# Project Modules

Проектный модуль содержит функциональность конкретного каталога.

## Пример: PflegeIndex

Модуль PflegeIndex может содержать:

* типы учреждений ухода;
* Kurzzeitpflege;
* Langzeitpflege;
* ambulante Pflege;
* специфические услуги;
* отраслевые фильтры;
* тексты и терминологию Pflege;
* проектные настройки бренда.

## Граница ответственности

Общая география относится к GeoCore.

Общая карточка объекта относится к DirectoryCore.

Общие контактные данные относятся к DataCore.

Поля, существующие только в сфере ухода, относятся к PflegeIndex.

---

# Правило определения модуля

Перед реализацией новой функции необходимо задать два вопроса.

## Вопрос 1

Может ли эта функция использоваться в следующем каталоге?

Если да, она должна находиться в Platform Core.

## Вопрос 2

Существует ли эта функция только из-за специфики текущей отрасли?

Если да, она должна находиться в Project Module.

---

# Направление зависимостей

Проектные модули могут использовать Platform Core.

Platform Core не должен зависеть от конкретного проектного модуля.

Правильное направление:

```text
PflegeIndex
→ DirectoryCore
→ DataCore
→ GeoCore
```

Неправильное направление:

```text
GeoCore
→ PflegeIndex
```

Ядро не должно знать, какой каталог использует его возможности.

---

# Архитектура публичных страниц

Для географического каталога используется следующая структура:

```text
Главная страница
→ Федеральная земля
→ Район
→ Город
→ Карточка объекта
```

Пример:

```text
/
→ /brandenburg.html
→ /brandenburg/dahme-spreewald.html
→ /brandenburg/luckau.html
→ /facility-name.html
```

Точная URL-структура может развиваться, но изменения существующих публичных URL требуют отдельного архитектурного решения.

---

# Внутренняя перелинковка

Страницы должны образовывать понятную иерархию.

```text
State
→ District
→ City
→ Facility
```

Допускаются дополнительные связи:

* объект → похожие объекты;
* объект → город;
* город → район;
* район → федеральная земля;
* категории → объекты;
* услуги → объекты.

Перелинковка должна быть полезна пользователю и понятна поисковым системам.

---

# Слой представления

Шаблоны интерфейса должны максимально использовать общие компоненты.

Примеры:

* карточка объекта;
* пагинация;
* breadcrumbs;
* список городов;
* блок похожих объектов;
* SEO-компоненты;
* сообщения об отсутствии результатов.

Проектные отличия должны передаваться через данные и настройки, а не через копирование целых шаблонов.

---

# Работа с базой данных

Изменения структуры базы данных выполняются только через миграции.

Нельзя вручную изменять production-базу без зафиксированной миграции или согласованной процедуры.

Каждая связь должна иметь понятную ответственность и необходимые индексы.

Преждевременное создание универсальной структуры данных не допускается.

---

# Тестирование архитектуры

Критически важная функциональность должна покрываться тестами.

В первую очередь тестируются:

* публичные маршруты;
* географическая принадлежность;
* фильтрация;
* пагинация;
* SEO-метаданные;
* структурированные данные;
* sitemap;
* доступность страниц;
* защита от некорректных данных.

Каждый новый модуль должен сохранять прохождение существующего набора тестов.

---

# Архитектурные изменения

Архитектурным считается изменение, которое влияет на:

* структуру модулей;
* направление зависимостей;
* публичные URL;
* модель данных;
* SEO;
* повторное использование;
* несколько будущих проектов.

Такие изменения должны быть обсуждены до реализации.

При необходимости решение фиксируется в каталоге:

```text
docs/decisions/
```

# Текущий статус реализации

На текущем этапе реализуется фундамент платформы.

Фундамент `DirectoryCore` реализован как независимый слой контрактов, value objects и read models. `Facility` остаётся моделью проекта PflegeIndex, `EntrySummary` используется только как read model списка, а `EntryRepository` реализуется проектными адаптерами.

Production-страницы `/pflegeheime.html` и `/brandenburg/{city}.html` получают основные списки через `ListEntries`, `PflegeEntryRepository` и project-specific presenter. Laravel paginator, URL generation, Blade и SEO остаются за пределами DirectoryCore. Region и district facility lists пока используют прямые Eloquent-запросы.

PHP-файлы внутри `app/Platform` защищены автоматическим dependency test. В Platform запрещены зависимости на Laravel/Illuminate, Eloquent, `App\Models`, `App\Projects`, Laravel facades и helpers.

Уже создана базовая географическая иерархия:

```text
State
→ District
→ City
→ Facility
```

GeoCore database foundation использует отдельные Country, State, District и Municipality records с AGS. Его Eloquent-модели и Brandenburg importer пока остаются infrastructure/application кодом Laravel, а не изолированным Platform-модулем. Обратная relation `GeoMunicipality::cities()` на project-модель `City` является известным архитектурным долгом и должна устраняться только в отдельной GeoCore-итерации.

PflegeIndex используется как первый проект для проверки архитектуры.

Остальные модули будут добавляться постепенно, только когда появится реальная потребность.

---

# Главная архитектурная цель

Следующий каталог должен запускаться не путём копирования PflegeIndex, а путём подключения нового проектного модуля к существующему ядру Directory Platform.
