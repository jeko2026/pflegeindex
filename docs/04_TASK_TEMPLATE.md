# 04_TASK_TEMPLATE.md

# Стандарт постановки задач

## Назначение документа

Все задачи для разработки должны использовать единый шаблон.

Это позволяет:

- уменьшить количество уточняющих вопросов;
- избежать неправильного понимания задачи;
- ускорить разработку;
- получать предсказуемый результат.

---

# Общий шаблон

## Название

Краткое название задачи.

---

## Цель

Что необходимо получить после выполнения задачи.

Описывается ожидаемый результат, а не способ реализации.

---

## Контекст

Какая функциональность уже существует.

Какие документы или решения необходимо учитывать.

Если задача связана с архитектурой, указываются соответствующие документы из папки docs.

---

## Требования

Список требований.

Например:

- использовать существующую архитектуру;
- не создавать дублирование;
- соблюдать соглашения Laravel;
- использовать существующие компоненты;
- сохранить совместимость.

---

## Ограничения

Что запрещено делать.

Например:

- без изменения структуры базы;
- без новых зависимостей;
- без изменения публичных URL;
- без изменения SEO;
- без изменения архитектуры.

---

## Тестирование

Какие проверки необходимо выполнить.

Например:

- Feature Tests
- Unit Tests
- Browser checks
- Проверка SEO
- Проверка JSON-LD
- Проверка Sitemap

---

## Ожидаемый отчёт

После выполнения задачи необходимо предоставить:

- что реализовано;
- какие файлы изменены;
- какие тесты выполнены;
- результат проверки;
- возможные улучшения.

---

# Пример задачи

## Название

Добавить страницы районов Brandenburg.

## Цель

Реализовать страницы районов с сохранением существующей архитектуры GeoCore.

## Контекст

Использовать:

- GeoCore
- существующие маршруты
- существующие Blade-компоненты

Не нарушать текущую структуру URL.

## Требования

- использовать существующие контроллеры при возможности;
- переиспользовать шаблоны;
- добавить необходимые тесты;
- использовать существующие SEO-компоненты.

## Ограничения

Запрещается:

- изменять миграции;
- изменять структуру базы данных;
- изменять существующие URL;
- изменять архитектуру GeoCore.

## Тестирование

Проверить:

- все новые страницы;
- breadcrumbs;
- JSON-LD;
- sitemap;
- существующие тесты.

## Итоговый отчёт

Предоставить:

- список изменённых файлов;
- результаты тестов;
- краткое описание реализации.

---

# Контрольный список перед отправкой задачи

Перед передачей задачи необходимо убедиться:

☐ цель сформулирована;

☐ ограничения указаны;

☐ архитектура не нарушается;

☐ понятно, что считается завершением задачи;

☐ известно, какие тесты необходимо выполнить.

---

# Контрольный список после выполнения

После завершения задачи необходимо проверить:

☐ задача выполнена полностью;

☐ тесты проходят;

☐ документация при необходимости обновлена;

☐ лишний код отсутствует;

☐ архитектура сохранена;

☐ решение готово к следующему этапу разработки.

---

# Главный принцип

Каждая задача должна быть максимально понятной ещё до начала реализации.

Чем точнее сформулирована задача, тем выше качество результата и тем меньше вероятность ошибок.
