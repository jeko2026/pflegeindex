# Основные принципы Directory Platform

## Назначение документа

Этот документ определяет основные принципы разработки платформы.

Любое архитектурное или техническое решение должно соответствовать данным принципам.

Если решение противоречит хотя бы одному из них, необходимо пересмотреть подход или обсудить его до начала реализации.

---

# Принцип 1. Концепция важнее кода

Любая разработка начинается с понимания задачи.

Сначала формируется идея.

Затем проектируется архитектура.

И только после этого начинается реализация.

Код является следствием архитектурных решений, а не наоборот.

---

# Принцип 2. Архитектура важнее скорости

Мы не стремимся написать код как можно быстрее.

Мы стремимся построить систему, которая останется понятной и расширяемой через годы.

Иногда более медленное решение сегодня позволяет избежать больших затрат в будущем.

---

# Принцип 3. Простота — главный критерий качества

Простая система почти всегда лучше сложной.

Мы избегаем:

- лишних уровней абстракции;
- преждевременной оптимизации;
- ненужных зависимостей;
- дублирования логики.

Каждый новый модуль должен уменьшать общую сложность платформы или, как минимум, не увеличивать её.

---

# Принцип 4. Повторное использование

Перед разработкой новой функциональности необходимо задать вопрос:

"Можно ли использовать этот модуль в следующем каталоге?"

Если ответ положительный, функциональность должна быть частью платформы.

Если ответ отрицательный — она относится к конкретному проекту.

---

# Принцип 5. Один источник истины

Каждая область ответственности имеет единственный источник данных.

Примеры:

- GeoCore — география;
- DirectoryCore — маршрутизация;
- DataCore — данные объектов;
- SEOCore — SEO.

Не допускается создание альтернативных механизмов, дублирующих существующую функциональность.

---

# Принцип 6. Обратная совместимость

Новые изменения не должны нарушать работу существующих функций.

Особое внимание уделяется:

- URL;
- SEO;
- структуре данных;
- публичным страницам;
- API (если появится).

Если изменение потенциально нарушает совместимость, оно должно быть отдельно согласовано.

---

# Принцип 7. Документация — часть продукта

Документация сопровождает разработку.

Изменение архитектуры должно сопровождаться обновлением соответствующих документов.

Документация должна отражать текущее состояние платформы.

---

# Принцип 8. Решения принимаются на уровне архитектуры

Если задача влияет на устройство платформы, сначала принимается архитектурное решение.

Только после этого начинается реализация.

Архитектура определяет код, а не наоборот.

---

# Принцип 9. Минимальный технический долг

Мы не откладываем известные проблемы без необходимости.

Если временное решение неизбежно, оно должно быть явно обозначено и внесено в ROADMAP.

---

# Принцип 10. Платформа развивается постепенно

Мы предпочитаем небольшие завершённые этапы.

Каждый этап должен:

- иметь понятную цель;
- быть протестирован;
- не ухудшать существующую архитектуру;
- завершаться документированием результата.

---

# Принцип 11. Единый стиль разработки

Весь проект должен восприниматься как единая система.

Независимо от времени написания или автора кода:

- структура должна оставаться последовательной;
- соглашения — едиными;
- архитектурные решения — согласованными.

---

# Главный принцип платформы

Любое новое решение должно усиливать платформу, а не только текущий проект.

Если существует возможность сделать решение универсальным без существенного усложнения системы, предпочтение отдаётся универсальному подходу.
