# Editorial & Contact Enrichment Model

## 1. Назначение и границы

Этот документ описывает безопасную редакционную модель для постепенного
улучшения карточек учреждений PflegeIndex и будущих каталогов Directory
Platform.

Документ является результатом архитектурного аудита. На этом этапе не
изменяются модели, миграции, база данных, контроллеры, Blade, публичные URL или
SEO-поведение.

Основные принципы:

- официальные исходные сведения и редакционные дополнения всегда различимы;
- наличие значения не считается подтверждением его актуальности;
- публикация требует источника, даты проверки и ответственного решения;
- автоматизация создаёт предложения или черновики, но не подменяет редактора;
- опубликованные URL и защищённые ручные контакты сохраняются;
- история должна позволять понять, кто, когда, почему и на основании чего
  изменил публичное значение;
- в карточку нельзя добавлять неподтверждённые услуги, медицинские обещания,
  рейтинги или вымышленные преимущества.

## 2. Текущее состояние

### 2.1 Что уже реализовано

Текущая таблица `facilities` объединяет импортированные базовые сведения,
редакционный контент и служебные поля проверки.

Администратор уже может редактировать:

- `description`;
- `phone`;
- `email`;
- `website`;
- `contact_source`;
- `contact_status`;
- `contact_locked`.

Для описаний существует отдельный процесс:

```text
description_draft
→ проверка текста, источников и даты
→ сохранение или публикация
→ description + description_sources + description_checked_at
```

Для контактов существует очередь `contact_suggestions`:

```text
parser result
→ pending suggestion
→ ручная проверка
→ accepted / rejected
→ обновление facility
```

Принятый контакт получает дату проверки и защиту `contact_locked`, поэтому
обычный повторный импорт не должен перезаписать вручную подтверждённые данные.

### 2.2 Ограничения текущей реализации

- официальные и редакционные значения физически находятся в одной записи;
- для контактов хранится одна общая ссылка-источник, а не provenance каждого
  поля;
- прямое редактирование facility не создаёт полноценную версионную историю;
- решение по parser suggestion сохраняется, но не заменяет общий audit trail;
- статус контакта применяется к набору контактов, а не отдельно к телефону,
  e-mail и website;
- `updated_at` показывает техническое обновление строки, но не причину,
  источник или редактора;
- отсутствуют часы работы, фотографии, права на изображения, редакционные
  заметки, дата следующей проверки и структурированные дополнительные контакты;
- `care_types` и `features` импортируются как JSON, но не имеют собственного
  редакционного статуса и источника;
- координаты предусмотрены текущим Quality Score, но отсутствуют в схеме
  `facilities`;
- наличие `description` само по себе не доказывает уникальность, редакционную
  проверку или актуальность текста.

## 3. Инвентаризация данных

### 3.1 Official data

| Данные | Текущие поля | Правило |
|---|---|---|
| Идентификатор исходной записи | `source_id` | Стабильная связь с исходным набором; не переиспользовать для другой организации |
| Наименование | `name` | Считать официальным только при подтверждённом происхождении исходного набора |
| Адрес | `postal_code`, `street`, `house_number`, `address` | Сохранять исходное значение и не угадывать отсутствующие части |
| Тип и сектор | `type`, `source_sector` | Официальная классификация источника, не маркетинговая категория |
| Населённый пункт | `city_id` и GeoCore-связь города | Географическая нормализация не должна менять исходное название учреждения |
| Классификационные списки | `care_types`, `features` | Сейчас импортируются; официальный статус допустим только при известном источнике каждого значения |

`slug` и публичный URL не являются официальными сведениями об учреждении. Это
техническая идентификация PflegeIndex, которую необходимо сохранять ради
обратной совместимости.

### 3.2 Editorial data

| Данные | Текущие поля | Текущее состояние |
|---|---|---|
| Публичное описание | `description` | Редактируется; может быть импортированным или опубликованным редакционным текстом |
| Черновик описания | `description_draft` | Не публикуется до решения редактора |
| Источники описания | `description_draft_sources`, `description_sources` | Список абсолютных HTTP(S)-адресов |
| Проверка описания | `description_draft_checked_at`, `description_checked_at` | Дата проверки источников |
| Признак AI assistance | `description_ai_assisted` | Прозрачность внутреннего процесса; AI не является источником |
| Контакты | `phone`, `email`, `website` | Редактируются вручную и могут поступать из parser suggestions |
| Источник контактов | `contact_source` | Одна ссылка для текущего набора контактов |
| Состояние контактов | `contact_status`, `contact_checked_at` | `verified`, `pending`, `not_found` или отсутствие статуса |
| Защита ручной работы | `contact_locked` | Не позволяет обычному импорту перезаписать подтверждённые контакты |
| Кандидаты контактов | `contact_suggestions.*` | Содержат найденные значения, источники, confidence и решение редактора |

### 3.3 Computed and system data

| Данные | Источник вычисления | Хранение |
|---|---|---|
| Canonical URL | Route + city slug + facility slug | Не должен становиться редакционным полем |
| Полный public URL | Laravel routing | Вычисляется при выводе |
| Форматированный телефон | `Facility::formattedPhone()` | Вычисляется из `phone` |
| Google Maps search URL | Адрес учреждения | Вычисляется при выводе, запрос отправляется только после клика |
| Data Quality Score | Наличие и валидность уже загруженных данных | Вычисляется динамически, не хранится |
| Trust badges | Выполненные критерии Quality Score | Вычисляются динамически |
| Related facilities | Город и текущая выборка | Вычисляются контроллером |
| Correction mailto | Имя учреждения + canonical URL | Вычисляется при выводе |
| `created_at`, `updated_at` | Laravel | Системные timestamps, не редакционная история |

### 3.4 Отсутствующие данные

Текущая модель не поддерживает полноценно:

- несколько телефонов, e-mail или сайтов с назначением и приоритетом;
- часы работы и исключения для праздников;
- фотографии, alt-тексты, авторство, лицензии и согласование прав;
- внутренние и публичные редакционные заметки;
- источник и статус каждого отдельного поля;
- автора прямого изменения facility;
- reviewer/publisher для каждого опубликованного набора изменений;
- причину изменения или отклонения;
- дату следующей проверки;
- архив опубликованных версий;
- структурированные конфликты между источниками;
- локализацию редакционного контента;
- подтверждённые координаты и их источник.

## 5. Целевая редакционная модель

Это концептуальная модель для будущего спринта, а не предложение немедленно
перенести текущие данные.

### 5.1 Editorial record

Одна редакционная единица должна описывать значение или логически связанный
набор значений:

| Поле | Назначение |
|---|---|
| `directory_entry_id` | Ссылка на запись конкретного каталога |
| `field_key` | Например `description`, `contact.phone`, `opening_hours` |
| `locale` | Язык контента, по умолчанию `de` |
| `value` | Типизированное текстовое или JSON-значение |
| `origin` | `official`, `editorial`, `external_verified`, `unknown` |
| `verification_state` | `unknown`, `needs_review`, `verified` |
| `publication_state` | `draft`, `in_review`, `published`, `archived` |
| `verified_at` | Когда источник был проверен |
| `verified_by` | Кто выполнил проверку |
| `published_at` | Когда версия стала публичной |
| `published_by` | Кто принял решение о публикации |
| `next_review_at` | План повторной проверки |
| `change_reason` | Краткая причина изменения |

Не все поля обязаны быть отдельными SQL-строками. Важно сохранить эти понятия и
не смешивать происхождение, проверку и публикацию в одном статусе.

### 5.2 Contact point

Контакт должен иметь:

- тип: `phone`, `email`, `website`;
- нормализованное значение;
- отображаемое значение;
- назначение: general, admission, management, emergency или другое
  утверждённое значение;
- приоритет;
- origin и verification state;
- собственный источник;
- `verified_at`, `verified_by`, `next_review_at`;
- active/withdrawn status;
- признак защиты от автоматического импорта.

Автоматический импорт может предложить новое значение, но не должен
перезаписывать protected contact point.

### 5.3 Opening hours

Часы работы следует хранить структурировано:

- timezone;
- weekday;
- один или несколько интервалов;
- тип часов: office, telephone, visiting;
- исключения и праздничные дни;
- пояснение для пользователя;
- источник и дата проверки.

Свободный текст допустим только как дополнение. Нельзя превращать общие часы
офиса в обещание круглосуточной доступности учреждения.

### 5.4 Media asset

Для фотографии необходимы:

- локальный контролируемый файл или утверждённое хранилище;
- alt text;
- подпись;
- автор/правообладатель;
- источник;
- основание использования или лицензия;
- дата проверки прав;
- статус draft/review/published/archived;
- размеры, MIME type и checksum;
- безопасная crop/focal-point информация.

Search snippets, Google Maps и изображения без подтверждённых прав не являются
допустимым источником публикации.

### 5.5 Editorial note

Заметки должны явно разделяться:

- `internal`: видит только редакция;
- `public`: нейтральное пояснение, прошедшее публикацию.

Внутренние заметки не должны попадать в Blade, API, sitemap, JSON-LD или логи
публичных запросов.

### 5.6 Source reference

Источник должен быть отдельным объектом provenance:

- полный HTTP(S) URL;
- publisher/owner;
- название страницы;
- тип источника: authority, facility, provider, public register;
- дата доступа;
- дата сведений, если она указана источником;
- поля, которые источник подтверждает;
- статус доступности;
- необязательный content hash для обнаружения изменений;
- комментарий редактора.

Provider homepage не должен автоматически считаться страницей конкретного
учреждения. Предпочтителен прямой официальный URL учреждения.

### 5.7 Change set and history

Каждое решение должно образовывать неизменяемое событие:

- entry;
- затронутые поля;
- previous/new value или безопасный diff;
- source references;
- actor;
- action;
- timestamp;
- reason;
- automated/manual origin;
- publication result.

История не должна содержать пароли, tokens, session data или неограниченный raw
HTML. Персональные данные редакторов показываются только авторизованным ролям.

## 6. Универсальные статусы

Предложенные названия смешивают две разные характеристики:

- `Official source` и `Editorial` описывают происхождение;
- `Verified`, `Needs review` и `Unknown` описывают состояние проверки.

Поэтому рекомендуется хранить две оси.

### 6.1 Origin

| Статус | Значение |
|---|---|
| `Official source` | Значение непосредственно получено из подтверждённого официального источника |
| `Editorial` | Значение сформулировано редакцией на основании указанных источников |

Дополнительно платформе потребуется технический origin `external_verified` для
контактов с официальной страницы учреждения, которые не являются государственными
данными.

### 6.2 Verification state

| Статус | Значение |
|---|---|
| `Verified` | Значение проверено человеком, источник доступен, дата зафиксирована |
| `Needs review` | Источник устарел, недоступен, конфликтует либо срок проверки истёк |
| `Unknown` | Происхождение или актуальность ещё не исследованы |

### 6.3 Правила переходов

```text
Verification:
Unknown
  → Needs review        создана задача проверки
  → Verified            значение сразу подтверждено человеком и источником

Needs review
  → Verified            редактор подтвердил значение и источник

Verified
  → Needs review        истёк срок, источник изменился или найден конфликт
  → Unknown             значение удалено и provenance невозможно восстановить

Origin новой версии:
Unknown
  → Official source     значение перенесено из официального источника
  → Editorial           значение сформулировано редакцией по источникам
```

Правила:

- автоматизация не переводит значение в `Verified`;
- заполненное поле не становится `Official source` без provenance;
- `Official source` не означает, что значение актуально бессрочно;
- редакционный текст не маркируется как официальный;
- origin существующей версии не меняется только из-за повторной проверки;
- изменение origin означает публикацию новой версии значения;
- отклонение предложения не меняет опубликованное значение;
- conflicting source создаёт review task, а не автоматическую замену;
- status применяется к полю или версии, а не ко всей карточке сразу;
- публикация и verification являются отдельными решениями.

## 7. Contact enrichment workflow

### 7.1 Source

Приоритет источников:

1. страница конкретного учреждения;
2. официальный сайт оператора с однозначной карточкой учреждения;
3. государственный или муниципальный реестр;
4. другой явно атрибутируемый первичный источник.

Search results, snippets, reviews и карты можно использовать только для
обнаружения кандидата, но не как окончательное доказательство.

### 7.2 Проверка

- URL проходит синтаксическую и server-fetch policy проверку;
- совпадают название, адрес, город и при необходимости PLZ;
- контакт найден на странице, а не только в snippet;
- фиксируется отдельный источник для каждого найденного значения;
- телефон нормализуется без угадывания цифр или добавочных номеров;
- e-mail приводится к безопасному отображению;
- website относится к учреждению, а не только к общей homepage;
- parser confidence помогает приоритизации, но не заменяет проверку.

### 7.3 Редактор

Редактор видит diff между опубликованными данными и предложением, источники,
дату проверки, confidence и предупреждения. Он может принять отдельные поля,
отклонить предложение или отправить его на дополнительную проверку.

### 7.4 Публикация

- применяется только утверждённый change set;
- операция атомарна;
- protected/manual значения не перезаписываются без явного решения;
- опубликованная версия получает actor, source и timestamp;
- URL, canonical и slug остаются неизменными;
- пустое новое значение не удаляет действующий контакт без отдельного решения;
- публичный cache очищается только после успешной публикации.

### 7.5 Повторная проверка

Периоды должны быть конфигурируемой редакционной политикой. Начальная рабочая
гипотеза:

- контакты: каждые 6 месяцев;
- часы работы: каждые 3–6 месяцев;
- описания: каждые 12 месяцев или при изменении источника;
- официальные сведения: после нового выпуска исходного набора;
- media rights: до истечения лицензии или разрешения.

Эти интервалы требуется утвердить владельцу; они не являются юридически
обязательными сроками.

### 7.6 История изменений

История должна отвечать на вопросы:

- какое значение было раньше;
- что опубликовано сейчас;
- кто принял решение;
- какой источник использовался;
- когда выполнена проверка;
- почему предложение принято или отклонено;
- можно ли безопасно восстановить предыдущую версию.

## 8. Роли

| Роль | Ответственность | Не должна делать |
|---|---|---|
| Importer / Parser | Создаёт кандидаты, нормализует и фиксирует источники | Публиковать автоматически |
| Researcher | Находит первичные источники и готовит предложения | Выдавать discovery source за подтверждение |
| Editor | Редактирует текст и контакты, объясняет изменения | Снимать protected flag неявно |
| Reviewer | Проверяет diff, источник, формулировки и права | Одобрять неподтверждённые claims |
| Publisher | Выпускает утверждённый change set | Менять данные вне зафиксированного набора |
| Data steward / Admin | Управляет политикой, конфликтами, rollback и доступами | Массово обходить audit trail |

В небольшом проекте один человек может иметь несколько ролей. Система всё равно
должна фиксировать, в какой роли совершено действие. Для массовых публикаций и
рискованных изменений рекомендуется принцип четырёх глаз.

AI может помогать с черновиком и классификацией, но:

- не является источником;
- не публикует самостоятельно;
- не придумывает услуги или характеристики;
- его участие маркируется во внутренней истории;
- итог проверяется человеком по первичным источникам.

## 9. Quality metrics

Все метрики должны иметь фиксированный denominator, правила валидности и дату
среза.

| Метрика | Рекомендуемая формула |
|---|---|
| Карточки с телефоном | Valid active phone / все опубликованные карточки |
| Карточки с website | Valid active HTTP(S) facility URL / все опубликованные карточки |
| Карточки с e-mail | Valid active e-mail / все опубликованные карточки |
| Карточки с уникальным описанием | Published, source-backed, reviewed, non-template description / все опубликованные карточки |
| Недавно проверенные карточки | Карточки хотя бы с одним актуальным verified field / все опубликованные карточки |

Дополнительные operational metrics:

- доля контактов с отдельным source reference;
- доля карточек с просроченной `next_review_at`;
- median age последней проверки;
- pending suggestions и их median waiting time;
- acceptance/rejection rate parser suggestions;
- количество конфликтов источников;
- количество protected fields, которые импорт попытался изменить;
- доля публикаций с reviewer, publisher и change reason;
- доля media assets с подтверждёнными правами и alt text.

Метрика «недавно проверено» должна использовать утверждённое окно, например 12
месяцев. Один технический `updated_at` не считается проверкой.

## 10. Требования к будущему редактору

### 10.1 Безопасность данных

- public draft isolation;
- role-based permissions;
- CSRF и server-side validation;
- URL validation и SSRF-safe fetch policy;
- ограничение типов и размеров upload;
- очистка metadata изображений при необходимости;
- неизменяемый audit trail;
- атомарная публикация и проверяемый rollback;
- отсутствие production auto-publish.

### 10.2 UX редактора

- поиск по имени, городу, PLZ и source ID;
- фильтры missing/stale/needs review;
- side-by-side diff;
- источник рядом с каждым полем;
- понятное разделение draft, verified и published;
- предупреждение о protected/manual значении;
- preview публичной карточки без публикации;
- bulk actions только для однородных проверенных change sets;
- keyboard-accessible интерфейс и видимые focus states;
- явный success/error result.

### 10.3 Редакционные правила

- нейтральный фактический немецкий язык;
- отсутствие медицинских рекомендаций и гарантий;
- отсутствие неподтверждённых рейтингов и преимуществ;
- точная атрибуция официальных данных;
- перефразирование источников вместо копирования;
- отдельное обозначение редакционных дополнений;
- обязательная проверка прав на фотографии;
- facility-specific source предпочтительнее provider homepage;
- опубликованные описания и protected contacts не перезаписываются bulk import.

## 11. Обратная совместимость и этапы внедрения

Будущая реализация должна быть поэтапной:

1. Зафиксировать field taxonomy, роли, статусы и retention policy.
2. Добавить audit/change-set слой без изменения public read path.
3. Перенести contact suggestions в field-level review workflow.
4. Подключить structured contacts и opening hours с fallback на текущие поля.
5. Добавить media workflow после утверждения storage и rights policy.
6. Перевести editor на versioned publication.
7. Только после parity-тестов переключить public read model.
8. Сохранять текущие facility fields до завершения совместимого перехода.

На каждом этапе:

- те же URL и canonical;
- тот же public content до явной публикации;
- отсутствие дополнительных SQL-запросов в циклах;
- rollback не требует изменения slug;
- protected contacts сохраняются;
- import остаётся идемпотентным;
- production database изменяется только отдельным backup-first deployment.

## 12. Повторное использование в Directory Platform

Универсальными могут быть:

- status/origin vocabulary;
- source reference;
- change set и audit event;
- workflow draft/review/publish/archive;
- contact point;
- opening hours;
- media rights metadata;
- quality metric contracts;
- роли и permission boundaries.

В PflegeIndex должны остаться:

- LASV и Pflegefonds attribution;
- термины Pflegeeinrichtung и care-specific taxonomy;
- правила допустимых медицинских формулировок;
- PflegeIndex Quality weights;
- немецкие тексты интерфейса и editorial policy;
- существующие маршруты, slugs и SEO metadata;
- конкретные интервалы повторной проверки после утверждения владельцем.

Платформенный слой не должен знать Eloquent-модель `Facility`, маршруты
PflegeIndex или названия LASV. Второй каталог должен подключать свои entry
adapter, field taxonomy, источники, роли и public presenter.

## 13. Критерии готовности будущей реализации

Редакционная инфраструктура считается готовой, когда:

- каждое публичное редакционное значение имеет origin;
- каждое verified value имеет источник, дату и actor;
- draft никогда не попадает в публичный ответ;
- parser не может публиковать без решения человека;
- protected values переживают повторный импорт;
- доступны field-level diff и история;
- отзыв публикации не меняет URL;
- фотографии без прав не публикуются;
- stale data автоматически становится задачей, но не удаляется автоматически;
- Quality metrics воспроизводимы одним и тем же набором правил;
- новый слой не изменяет canonical, JSON-LD, Open Graph или sitemap без
  отдельного SEO-решения.

## 14. Рекомендованный следующий этап

Следующий этап должен быть ограничен проектированием field taxonomy и
versioned change-set schema с прототипом read-only migration plan.

До утверждения этого плана не следует добавлять отдельные поля для фотографий,
часов работы или заметок непосредственно в `facilities`: это снова смешает
официальные, редакционные и служебные данные и усложнит повторное использование
Directory Platform.
