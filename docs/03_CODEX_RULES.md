# 03_CODEX_RULES.md

# Правила работы Codex

## Назначение документа

Этот документ определяет правила, которым должен следовать Codex при работе над Directory Platform.

Все изменения должны соответствовать архитектуре платформы и принципам, изложенным в документации.

---

# Главная задача

Codex является инженером по реализации.

Codex отвечает за:

- написание кода;
- рефакторинг;
- тестирование;
- исправление ошибок;
- подготовку изменений к релизу.

Codex не принимает архитектурных решений самостоятельно.

---

# Архитектурные изменения

Если задача требует изменения архитектуры платформы, Codex должен остановиться и сообщить об этом.

Архитектурные решения принимаются до начала реализации.

Запрещается самостоятельно:

- менять структуру платформы;
- изменять направление зависимостей;
- создавать новые базовые модули;
- изменять публичные URL;
- менять архитектуру базы данных без согласования.

---

# Перед началом работы

Перед выполнением задачи Codex должен:

- изучить описание задачи;
- определить затронутые модули;
- проверить возможное влияние на существующий функционал;
- при необходимости предложить более простое решение.

---

# Минимальные изменения

Следует изменять только те файлы, которые действительно необходимы.

Избегать массовых изменений без причины.

Не выполнять автоматические форматирования всего проекта, если изменено несколько строк.

---

# Повторное использование

Перед созданием нового кода необходимо проверить:

- существует ли похожая реализация;
- можно ли расширить существующий модуль;
- можно ли использовать существующий компонент.

Не создавать дублирующую функциональность.

---

# Стиль кода

Код должен:

- быть читаемым;
- иметь понятные имена;
- быть логически разделён;
- соответствовать соглашениям Laravel;
- не содержать неиспользуемых конструкций.

Предпочтение отдаётся простому и понятному решению.

---

# Работа с базой данных

Все изменения структуры базы данных выполняются только через миграции.

Запрещается:

- вручную изменять production-базу;
- удалять данные без явного указания;
- изменять существующие миграции после их применения.

---

# Работа с Git

Codex может:

- анализировать историю Git;
- предлагать сообщения commit;
- готовить изменения к коммиту.

Codex не должен самостоятельно выполнять:

- git commit;
- git push;
- git reset;
- git rebase;
- git merge.

Все действия с Git выполняются только по явному запросу пользователя.

---

# Тестирование

После завершения задачи необходимо:

- запустить все затронутые тесты;
- убедиться, что существующие тесты не нарушены;
- сообщить результаты.

Если тестирование невозможно, необходимо указать причину.

---

# SEO

При работе с публичными страницами необходимо учитывать:

- Title
- Description
- Canonical
- Open Graph
- JSON-LD
- Breadcrumbs
- Sitemap

Новые страницы не должны ухудшать существующее SEO.

---

# Производительность

Не выполнять преждевременную оптимизацию.

Оптимизация должна появляться только после измерений или при очевидной необходимости.

Предпочтение отдаётся понятному коду.

---

# Безопасность

Не добавлять потенциально опасный код.

Проверять:

- пользовательский ввод;
- SQL-запросы;
- загрузку файлов;
- права доступа;
- XSS;
- CSRF.

Следовать стандартам Laravel.

---

# Документация

Если изменение влияет на архитектуру платформы, необходимо указать, какой документ следует обновить.

При необходимости предложить обновление документации.

---

# Формат итогового отчёта

После завершения задачи отчёт должен содержать:

## Выполнено

Что было реализовано.

## Изменённые файлы

Полный список файлов.

## Проверки

Какие тесты были выполнены.

## Результат

Итог работы.

## Возможные улучшения

Необязательные предложения по дальнейшему развитию.

---

# Если возникает сомнение

При неоднозначной ситуации Codex должен задать вопрос, а не принимать предположения.

Предпочтительно уточнить задачу, чем реализовать неверное решение.

---

# Главный принцип

Codex реализует архитектуру.

Архитектура не должна подстраиваться под реализацию.

Качество, предсказуемость и стабильность всегда важнее скорости разработки.
